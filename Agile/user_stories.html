User Stories

What are user stories?

User stories are the written requirements used in many of the popular agile processes. These are very different than the traditional requirements that are used by the traditional heavyweight models. The heavy weight models normally use the requirements document outlined in IEEE 830, which outlines the guidelines for creating good software requirements.

User Stores are comprised of one sentence features that are written by the users of the software. According to Mike Cohn, users stories are made up of three components:

  • a written description of the story used for planning and as a reminder
  • conversations about the story that serve to flesh out the detials of the story
  • tests that convey and document details and that can be used to determind when a story is complete

Here is an example of a user story for a resume sharing website.

  • A user can view a job description
  • A user can view a job's salary range
  • A user can view the location of a job

This user story is one sentence that should have been written by a user or purchaser of the software. They explain features of the software that the users can perform. Each user story is limited to one feature. As you can see these stories are just the features. How do the other two components work?

Typically these user stories are written on notecards. The one sentence feature appears on the card. Also, on the card are notes from the conversation that is held about this requirement. The conversations can be held between the users and developers or between developers themselves. On the back of the cards there are acceptance test descriptions that need to be met before the feature can be marked as complete.

Conversations

The conversations are recorded below the user story. These conversations help to clarify ambiguities that can arise when it comes to analyze and design this requirement. For this proposed resume sharing website the job posters will need to pay for the job postings. So this user story is created.

  • A user can view information about each job that is matched in the search

This user story is fairly vague, but through conversations with the users and the software developers notes can be created.

  • "Marco says show description, salary, and location"

So the user of the software that created the user story explained what he wanted through conversations with the developers. The developers note these conversations so that the user story can be an ongoing record. This is helpful for when the user story is ready to be analyzed and designed.

Testing

On the back of the card are the requirements that need to pass before the feature is called complete. These come through conversations with the customer and users. For the resume sharing website this user story is written down:

  • A user can pay for posting jobs by credit card.

Through conversations the following tests are created with post conditions.

  • Test with Visa, Mastercard, and American Express (pass)
  • Test with Diner's Club (fail)
  • Test with Visa debit card (pass)
  • Test with good, bad and missing card ID numbers from the back of the card.
  • Test with expired cards.
  • Test with different purchase amounts (including one over the card's limit).

Epic Stories

Sometimes user stories can be very large and ambiguous. If the user stories can be broken up into smaller user stories this is called an epic story. For instance, this user story for the resume sharing website is considered an epic:

  • A user can search for a job

This user story is fairly vague and can be broken into other stories such as these:

  • A user can search for jobs by attributes like location, salary range, job title, company name, and date that the job was posted.
  • A user can view information about each job that is matched in the search
  • A user can view detailed information about a company that has posted a job.

With the epic story broken down into smaller stories it becomes more manageable to complete.

What user stories aren't?

IEEE 830

The guidelines that are represented in IEEE 830 do not represent what user stories are. The requirements that come out of the look like this:

  • The system shall allow a company to pay for a job posting with a credit card.
    • The system shall accept Visa, MasterCard and American Express cards.
    • The system shall charge the credit card before the job posting is placed on the site.
    • The system shall give the user a unique confirmation number

This type of requirements document requires all of the requirements to be completed before any analysis and design takes place. There is too much information placed in this document at the begining and is fairly rigid in terms of changing requirements. User stories are similar in the fact that they are requirements, but there is not as much detail placed in a user story. This is because user stories are placeholders for requirements. When the requirement needs to be completed a conversation takes place where all of the details can come from the user and developers, instead of coming from the beginning.

Use Cases

Use cases were popularized in the 1990s when the unified process became a popular software development lifecycle model. Use cases and user stories are similar in that they describe features. A use case contains some of the same information as a user story, but has much greater detail than a user story. They are normally created during the analysis and design phase of a software requirement.

For instance, a use case that describes an email feature might look like this:

Use Case Title: Compose and send email message
Main Success Scenario:

  1. User selects the "New Message" menu item.
  2. System presents the user with the "Compose New Message" dialog.
  3. User edits email body, subject field and recipient lines.
  4. User clicks the Send button.
  5. System sends the message.

The scope of a use case is much larger than that of a user story. A use case might cover a large requirement such as an epic user story, but they might even be larger than that. User stories are meant to be place holders for conversations between the developers and the users, so a user story will never be large since the conversation is not supposed to end throughout the entire development process. Use cases also last the entire project. User stories are meant to be used for a single sprint.

Bad Stories (Implementation Details)

There are some requirements that do not represent valued features because they represent implementation details. These, on occasion, could be user stories that exist in an application when it is required. For instance, the user might be a developer that will be interfacing with what you create, so it is important is it written in a certain language or in a certain way. Examples of these stories are below:

  • The software will be written in C++
  • The program will connect to the database through a connection pool.

These stories are normally considered bad because they do not represent a discussion held between the user and developers about valued features. Normally the users do not care how it is programmed as long as the features work as expected. The stories should not contain any implementation details either as they should appear in the notes.

How to write user stories?

Writing user stories is a very difficult process between users and developers. The requirements gathering process for any software project is probably the hardest phase. Requirements are hard to get because users do not know what they want. Mike Cohn, the author of User Stories Applied: For Agile Software Development, suggests that there are different techniques to gathering requirements.

  • User Interviews
    • Interview real users that fill different roles whenever possible
    • Ask open ended questions
  • Questionnaires
    • Can be helpful when trying to get information from a large group of people
    • This technique should supplement another technique
  • Observation
    • One of the best ways to improve the user experience is to watch the users use the applications
    • Unfortunately this can be very difficult to setup because the users are annonymous
    • This really only works when the users are in house
  • Story-Writing workshops
    • A meeting that contains developers, users, and other stake holders.
    • Participants write as many users stories as possible without setting priorities
    • These types of meetings can be held before every iteration
    • Some basic prototyping (workflows) can occur during these meetings

It might be helpful to use the following format when writing user stories. This template of writing user stories shows how the features are valued to the users. It is borrowed from Extreme Programming:

  • I as a (role) want (feature) so that (business value).
    • The roles are the types of users that access your application
    • The feature is what the user wants
    • The business value is why the user wants it.

Here are some tips for writing good stories

  • Keep the user interface details out of the user story.
    • The user should not care about how it looks, they only worry about the functionality. The UI is considered an implementation detail and should not be added to the user story.
  • Write for one user instead of groups
  • Include roles in the stories
  • Write in active voice
  • Put constraints on the cards
    • These constraints are not features but they must be obeyed
    • For instance "The system must support peak usage of up to 50 concurrent users." is an example of a constraint card.
  • Don't forget the purpose
    • These story cards are reminders to discuss the features when it comes time to develop them

Once the user stories are gathered, notes from the conversations about these cards are noted. These can be pursued further when they are ready to be analyzed. Since the stories are not analyzed right away the notes from the conversations are important.

It is also important to note that acceptance tests appear on the back of the card. Here are a few tips about testing:

  • Write your tests before you code.
  • Have the user write the tests

Why use user stories?

You may be asking yourselves why should we go through the process of writing user stories as requirements when there are already standards available? Mike Cohn has outlined many reasons and they will be explored more in depth.

  • Verbal Communication
    • Since user stories are placeholders for conversations about requirements, developers must communicate the needs of the users very often.
    • Helps to reduce ambiguity when it comes to the requirements
  • Comprehensible
    • Both users and developers should be able to understand the user stories.
    • The users know exactly what they should be getting.
  • Planning
    • User stories are the right size for planning.
    • IEEE 830 documents are too long and detailed when it comes to prioritizing requirements
    • Use Cases and Scenarios are too big to manage because they contain many requirements
  • Iterative Development
    • Not all of the user stories need to be completed before development begins
    • At the beginning of each iteration the user stories are reorganized and prioritized
    • The most important are chosen for development and the rest are put back for later iterations
    • Details emerge during the iterations instead of at the beginning of the development
  • Deferring Detail
    • User stories are placeholders for conversations about the requirements.
    • The analysis and design are done at the time of completion instead of at the beginning
    • IEEE 830 documents require analysis and design to be completed before development can begin.
  • Opportunistic design
    • It is impossible to get all of the requirements at the beginning of the project.
    • Software is not developed in a linear path
    • User stories allow for the creation of software by utilizing the current priorities.
  • Participatory Design
    • There is participation between users, developers and other stake holders throughout the entire process.
